# Spec Engineering：AI 工程方法论

> **版本**: v0.1.0 (2026-03-02)
> **状态**: 草案（从 OwlClaw 实践中提炼，待社区验证）
> **许可证**: CC BY-SA 4.0
> **一句话定义**: 人在 spec 层决策，AI 在代码层执行——半自动 AI 工程的最佳平衡点。

---

## 〇、为什么需要 Spec Engineering

### Vibe Coding 的问题

"Vibe Coding"（直觉式编程）是 2025-2026 年 AI 编程的主流模式：开发者在 Cursor/Claude Code 中用自然语言描述需求，AI 直接生成代码。这对原型和小项目有效，但在生产级软件中暴露出系统性问题：

| 问题 | 表现 | 根因 |
|------|------|------|
| 上下文丢失 | 多轮对话后 AI 忘记之前的决策 | 无持久化的设计文档 |
| 架构漂移 | AI 每次生成的架构不一致 | 无架构约束文档 |
| 需求偏移 | 实现与需求渐行渐远 | 无可追溯的需求文档 |
| 质量退化 | 代码越改越烂（"The Rot"） | 无验收标准和质量门控 |
| 协作困难 | 多个 AI Agent 互相覆盖 | 无并行协作规范 |
| 预算失控 | AI 反复试错消耗大量 token | 无明确的任务边界 |

### Spec Engineering 的核心理念

> **不要让 AI 猜你想要什么，用 spec 告诉它。**
> **不要让 AI 做架构决策，人来做。**
> **不要让 AI 单打独斗，给它一个团队。**

Spec Engineering 是一套**半自动 AI 工程方法论**，核心原则：

1. **人决策，AI 执行** — 人在 spec 层做业务和架构决策，AI 在代码层执行实现
2. **文档即契约** — spec 文档是人和 AI 之间的契约，也是 AI Agent 之间的契约
3. **循环驱动** — 不是一次性生成，而是多轮 spec 循环持续推进
4. **质量内建** — 验收标准写在 spec 里，不是事后补测试
5. **并行安全** — 多个 AI Agent 可以同时工作，通过物理隔离避免冲突

## 一、三层文档结构

每个功能或重大变更必须有三层文档：

```
.specs/{feature-name}/
├── requirements.md    # 需求文档（WHAT — 要做什么）
├── design.md          # 设计文档（HOW — 怎么做）
└── tasks.md           # 任务文档（WHO/WHEN — 谁来做、何时做）
```

### 为什么是三层，不是两层

大多数竞品（Kiro、Spec Kit、CodeGuide）只有 PRD + tasks 两层。缺少 design 层会导致：

- AI 自行做架构决策（通常是错的）
- 多个 AI Agent 对同一功能做出不同的架构选择
- 重构时没有架构文档可参考

**design.md 是架构决策的载体**，它回答"怎么做"而不是"做什么"。这一层必须由人来写（或人审核 AI 的草稿）。

### 三层的职责边界

| 文档 | 职责 | 写作者 | 内容重点 | 禁止包含 |
|------|------|--------|---------|---------|
| **requirements.md** | 定义做什么 | 产品/业务/人工 | User Story、验收标准、约束 | 技术实现细节、文件路径 |
| **design.md** | 定义怎么做 | 架构师/人工 | 架构图、数据模型、API、错误处理 | 任务清单、需求描述 |
| **tasks.md** | 定义执行计划 | 开发/AI 辅助 | 按阶段的任务清单、验收检查 | 需求描述、架构设计 |

### 文档关系

```
Requirements (需求) — 人工编写/审核
    ↓ 驱动
Design (设计) — 人工编写/审核
    ↓ 指导
Tasks (任务) — AI 可辅助生成，人工审核
    ↓ 执行
Code (代码) — AI 执行，人工审核
```

### tasks.md 模板

```markdown
# Tasks: [功能名称]

> **状态**：未开始/进行中/已完成
> **预估工作量**：X-Y 天
> **最后更新**：YYYY-MM-DD

## 进度概览
- **总任务数**：N
- **已完成**：N
- **进行中**：N
- **未开始**：N

## 1. [阶段名称]（X 天）
### 1.1 [任务名称]
- [ ] 1.1.1 子任务 1
- [ ] 1.1.2 子任务 2

## N. 验收清单
### N.1 功能验收
### N.2 测试验收

## N+1. 依赖与阻塞

## N+2. 风险
```

### 关键原则

- **凡设计即交付**：写入 tasks.md 的任务视为承诺交付
- **验收可执行**：验收标准应可被自动化执行（pytest、CLI 命令、脚本）
- **集成点显式写出**：涉及组件调用的任务必须写明"何时、何处调用"
- **收口以自动化验收为准**：收口 = 可执行验收通过

## 二、Spec 循环

Spec 循环是将 spec 文档中的 `[ ]` 通过实现+验收变为 `[x]` 的持续过程。

### 循环流程

```
1. Sync — 同步最新代码，确认任务分配
   ↓
2. Check — 检查未完成项，健康检查
   ↓
3. Plan — 选择本轮批次（1-3 个 task）
   ↓
4. Do — 对每个 task：
   ├─ Act（实现）：编写代码
   ├─ Observe（观察）：执行测试
   ├─ Reflect（反思）：结果符合预期？
   └─ Verify（验收）：通过 → 打勾；失败 → 分析原因
   ↓
5. Doc — 更新进度文档，记录检查点
   ↓
6. Commit — 提交本轮变更
   ↓
7. Evaluate — 是否继续？回到 1
```

### 核心规则

| 规则 | 说明 |
|------|------|
| **批次控制** | 单次循环最多 3 个 task，避免上下文溢出 |
| **连续执行** | 一个 task 完成后，若下一个可执行，接着做，不先汇报 |
| **失败处理** | 验收失败 → 分析原因 → 设计问题回退 design.md / 实现问题回滚代码 / 依赖问题标记 blocked |
| **健康检查** | 同一 task 连续失败 3 次 → 标记 blocked；单个 spec 循环 >10 轮无进展 → 人工介入 |
| **每轮提交** | 每轮结束必须 git commit，便于回溯 |
| **资源释放** | 测试执行后必须确保子进程已退出，防止内存泄漏 |

### 进度追踪

使用一个全局的**功能清单**文件（如 `SPEC_TASKS_SCAN.md`）作为单一真源：

```markdown
## 功能清单

| 功能 | Spec | 状态 | 进度 |
|------|------|------|------|
| 用户认证 | auth | ✅ 完成 | 5/5 |
| 数据导入 | data-import | ⏳ 进行中 | 3/7 |
| 报表生成 | reports | ❌ 未开始 | 0/4 |

## 检查点

| 字段 | 值 |
|------|---|
| 最后更新 | 2026-03-02 |
| 当前批次 | data-import Task 4-5 |
| 已完成项数 | 8/16 |
| 下一待执行 | data-import Task 4 |
| 阻塞项 | 无 |
| 健康状态 | 正常 |
```

## 三、多 Agent 协作

### 物理隔离模型

使用 Git Worktree 实现多 AI Agent 物理隔离：

```
project/              ← 主 worktree（main）— 人工 / 统筹
project-review/       ← 审校 worktree — AI 审校（技术经理角色）
project-agent-1/      ← 编码 worktree 1 — AI 编码
project-agent-2/      ← 编码 worktree 2 — AI 编码
```

所有目录共享同一个 `.git` 仓库，但文件系统完全独立。

### 角色定义

| 角色 | Worktree | 职责 |
|------|----------|------|
| **统筹** | 主 worktree | 任务分配、合并操作、冲突解决、架构决策 |
| **审校** | 审校 worktree | spec 一致性审核、代码质量把关、测试验证 |
| **编码** | 编码 worktree | 功能实现、测试编写、spec 循环推进 |

### 协作规则

1. **只做分配给自己的 spec/模块**，不越界
2. **两个编码 worktree 必须工作在不同的 spec**，避免合并冲突
3. **编码分支必须经过审校**才能合并到 main
4. **每轮工作结束必须 commit 所有变更**，工作目录必须干净
5. **启动时同步最新代码**：`git merge main`

### 契约先行规则

当多个 Agent 并行开发同一系统的不同层（如前端+后端）时：

- **必须先由一方输出共享契约文件并 commit 到 main**
- 另一方从契约生成代码/类型定义
- **禁止两端各自独立实现后再对齐**
- 契约形式：OpenAPI Schema / JSON Schema / Protocol 接口定义

### 合并流程

```
编码 worktree commit
  ↓
审校 worktree Review Loop（Scan → Review → Verdict）
  ↓ APPROVE
审校 worktree 合并到 review 分支并运行测试
  ↓ 测试通过
人工将 review 分支合并到 main
  ↓
各 worktree git merge main 同步
```

## 四、统筹循环

统筹循环是管理多个 AI Agent 协作的流程，由人工或统筹 AI 执行。

### 循环流程

```
1. Status — 获取全局状态
   - 各 worktree 的分支状态
   - 各 Agent 的当前任务和进度
   ↓
2. Merge — 合并已审核通过的变更到 main
   ↓
3. Sync — 同步 main 到各子 worktree
   - 检查工作状态：IDLE/DONE → merge；WORKING → 跳过
   ↓
4. Assign — 检查各编码 worktree 是否完成当前 spec
   - 完成 → 分配下一个 spec
   - 遵守契约先行规则
   ↓
5. Unblock — 评估 blocked 项是否可解锁
   ↓
6. Commit — 提交本轮变更
   ↓
7. Report — 输出状态总览表
```

### 工作状态标记

| 状态 | 含义 | 统筹行为 |
|------|------|---------|
| `IDLE` | 空闲，无活跃会话 | 可自由 merge/分配 |
| `WORKING` | 正在执行，可能有未提交改动 | **跳过 merge**，避免冲突 |
| `DONE` | 本轮完成并已 commit | 正常 merge |

## 五、半自动决策模型

### 决策分层

| 层次 | 决策者 | 内容 | 工具 |
|------|--------|------|------|
| **业务需求** | 人 | 做什么、为谁做、优先级 | requirements.md |
| **架构设计** | 人 | 技术选型、数据模型、API 设计 | design.md |
| **任务拆解** | 人 + AI | 实施步骤、验收标准 | tasks.md |
| **代码实现** | AI | 编写代码、测试 | Cursor / Claude Code / Codex CLI |
| **质量审核** | AI + 人 | spec 一致性、代码质量 | Review Loop |
| **合并决策** | 人 | 何时合并、冲突解决 | 统筹循环 |

### 为什么不全自动

全自动 AI 编码（如 SpecWeave 的 `/sw:auto`）在以下场景失败：

- **架构决策**：AI 倾向于选择最简单的方案，不考虑长期维护
- **业务规则**：AI 无法理解隐含的业务约束
- **安全边界**：AI 可能引入安全漏洞而不自知
- **成本控制**：全自动执行可能消耗大量 token 在错误方向上

半自动模型的优势：**人在高价值决策点介入（spec 层），AI 在低价值重复工作上执行（代码层）。**

## 六、适用场景

| 场景 | 是否适用 | 原因 |
|------|---------|------|
| 生产级软件开发 | 适用 | 需要架构约束和质量门控 |
| 多人/多 Agent 协作 | 适用 | Worktree 隔离 + 契约先行 |
| 长期维护的项目 | 适用 | spec 文档是持久化的知识 |
| 快速原型/hackathon | **不适用** | 过重，直接 vibe coding 更快 |
| 一次性脚本 | **不适用** | 不需要 spec |
| 纯前端 UI 调整 | **部分适用** | 简化版（只用 tasks.md）即可 |

## 七、与现有工具的关系

Spec Engineering 不是一个工具，而是一套方法论。它可以和以下工具组合使用：

| 工具 | 在方法论中的角色 |
|------|----------------|
| **Cursor / Continue.dev** | 交互编码（人在电脑前） |
| **Claude Code / Codex CLI** | 自主编码（AI 执行 spec 循环） |
| **GitHub Spec Kit / Kiro** | 可用于辅助生成 spec 初稿，但需人工审核 |
| **LiteLLM** | 模型路由（本地+云端混合） |
| **Langfuse** | 可观测（追踪 AI 调用成本和质量） |
| **Git Worktree** | 多 Agent 物理隔离 |

## 附录 A：从 Vibe Coding 到 Spec Engineering 的迁移路径

```
阶段 0：Vibe Coding（现状）
  你在 Cursor 里和 AI 聊天，AI 直接写代码
  ↓
阶段 1：加 tasks.md（最小改动）
  每个功能写一个 tasks.md，列出要做的事
  AI 按 tasks.md 执行，不再随意发挥
  ↓
阶段 2：加 requirements.md + design.md（三层完整）
  需求和架构决策持久化
  AI 不再自行做架构选择
  ↓
阶段 3：加 Spec 循环（持续推进）
  不再一次性生成，而是多轮循环
  每轮有验收、有检查点、有健康检查
  ↓
阶段 4：加多 Agent 协作（团队级）
  Worktree 隔离 + 审校 + 统筹
  一个人管理多个 AI Agent 并行开发
```

**建议**：从阶段 1 开始，每个阶段验证有效后再进入下一阶段。不要一步到位。
