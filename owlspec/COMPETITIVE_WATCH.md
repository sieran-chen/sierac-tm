## 一、竞品全景图

### 按"业务表格 → AI 代码"完整链路分段

```
业务表格（Excel / 飞书 / 钉钉表格）
    │
    ├─ 【表格分析】 ← 只做数据分析，不提取业务规则
    │   ├─ ChatExcel / 表答（自然语言操作表格）
    │   ├─ Quadratic / Excelmatic（AI 表格分析）
    │   └─ 勤哲 Excel 服务器（Excel → 管理系统，闭源）
    │
    ├─ 【表格 → Spec】 ← OwlSpec 的独特入口（无竞品）
    │
    ↓
PM 的业务需求（结构化 / 模糊）
    │
    ├─ 【需求结构化】 ← OwlSpec 的主战场
    │   ├─ CodeGuide（idea → PRD/spec，一次性）
    │   ├─ PM33（AI PRD + 优先级评分）
    │   ├─ VibeVibe.cn（教程，不是工具）
    │   └─ Claude Code for PMs（教 PM 用 AI，不是写 spec）
    │
    ↓
结构化 Spec
    │
    ├─ 【Spec → Code】 ← 已有成熟方案
    │   ├─ AWS Kiro（最成熟，GA 2025.11）
    │   ├─ GitHub Spec Kit（开源，2026.2）
    │   ├─ OpenSpec（开源，中国开发者社区）
    │   ├─ ZenCoder Zenflow（商业）
    │   └─ SpecWeave（开源）
    │
    ↓
AI 生成代码
    │
    ├─ 【AI 编码工具】 ← 基础设施层
    │   ├─ Cursor（IDE）
    │   ├─ Claude Code（CLI）
    │   ├─ OpenAI Codex CLI
    │   └─ Continue.dev（开源 IDE 插件）
    │
    ↓
可运行软件
```

## 二、重点竞品分析

### Tier 0：表格分析工具（相邻赛道，非直接竞争）

| 产品 | 做了什么 | 没做什么 | OwlSpec 差异 |
|------|---------|---------|-------------|
| **ChatExcel** | 自然语言操作表格 | 不提取业务规则 | OwlSpec 从表格提取规则生成 spec |
| **Quadratic** | AI 表格分析（多数据源） | 分析数据不分析业务逻辑 | OwlSpec 关注业务逻辑不关注数据 |
| **勤哲 Excel 服务器** | Excel → 管理系统 | 闭源平台锁定 | OwlSpec 输出通用 spec，不锁平台 |
| **PM33** | AI PRD + 优先级评分 | 不接受表格输入 | OwlSpec 从表格开始 |

**关键判断**：表格分析工具做的是"让表格更好用"，OwlSpec 做的是"从表格中提取业务知识，生成软件 spec"。赛道不同。

### Tier 1：大厂产品（直接竞争 Spec → Code 段）

| 产品 | 公司 | 定位 | 业务专家/PM 侧能力 | OwlSpec 差异 |
|------|------|------|------------------|-------------|
| **Kiro** | AWS | Spec-driven IDE | 自然语言 → EARS 需求 | PM 看不懂 EARS；不接受表格输入 |
| **Spec Kit** | GitHub | 开源 spec 工具 | 与 Copilot/Claude Code 集成 | 无 PM 侧模板；无表格解析 |
| **Copilot Workspace** | GitHub | AI 驱动开发 | Issue → Plan → Code | 无三层文档；无表格入口 |

**关键判断**：大厂产品全部面向开发者，没有"从表格开始"的入口。

### Tier 2：独立产品

| 产品 | 定位 | PM 侧能力 | OwlSpec 差异 |
|------|------|----------|-------------|
| **CodeGuide** | idea → PRD/spec | 一次性生成 PRD | 无表格输入、无迭代、无多人协作 |
| **ZenCoder Zenflow** | Spec 工作流 | 商业产品 | 无表格入口；价格壁垒 |
| **OpenSpec** | 开源 spec 管理 | 面向开发者 | 无 PM 侧；无表格解析 |

### Tier 3：方法论/教程

| 产品 | 定位 | 与 OwlSpec 关系 |
|------|------|----------------|
| **VibeVibe.cn** | 教 PM 用 AI 编码 | 教程不是工具；可参考其内容策略 |
| **Claude Code for PMs** | 教 PM 用 Claude Code | 教写代码不教写 spec；互补 |
| **CodingDiary "Vibe to Spec"** | 个人实践分享 | 验证了 Vibe → Spec 两阶段模式 |

## 三、差异化矩阵

| 维度 | Kiro | Spec Kit | CodeGuide | OpenSpec | ChatExcel | **OwlSpec** |
|------|------|----------|-----------|---------|-----------|-------------|
| 表格解析→Spec | 无 | 无 | 无 | 无 | 无 | **有** |
| 业务专家友好 | 低 | 低 | 中 | 低 | 高 | **高** |
| 三层文档 | 有（EARS） | 无 | 无 | 无 | 无 | **有（业务可读）** |
| Spec 循环 | 无 | 无 | 无 | 有 | 无 | **有（含健康检查）** |
| 多 Agent 协作 | 无 | 无 | 无 | 无 | 无 | **有** |
| 中文原生 | 无 | 无 | 无 | 部分 | 是 | **完全** |
| 开源 | 否 | 是 | 否 | 是 | 否 | **是** |
| 迁移路径 | 无 | 无 | 无 | 无 | 无 | **有（双入口）** |

### OwlSpec 的独特价值

1. **业务表格作为需求起点** — 没有竞品做"表格 → spec"，这是独家入口
2. **业务专家友好** — 不需要学新工具，上传自己的 Excel 即可
3. **PM 侧需求结构化** — 竞品全部面向开发者
4. **渐进式双入口迁移** — 从表格开始（业务专家）或从需求描述开始（PM/开发者）
5. **多 Agent 协作** — Worktree 隔离 + 审校 + 统筹
6. **中文原生** — 文档、模板、培训全中文
7. **实战验证** — 从 OwlClaw 开发中提炼，不是纸上谈兵

## 四、需要持续关注的动向

| 信号 | 影响 | 应对 |
|------|------|------|
| Kiro/Spec Kit 增加表格输入 | 直接竞争核心差异化 | 加速 PoC，建立先发优势 + 中文生态 |
| ChatExcel 增加 spec 输出 | 从表格侧切入 | 深耕"表格→spec→代码"全链路，不只做表格分析 |
| OpenSpec 增加 PM 模板 | 差异化缩小 | 深耕培训/咨询 + 表格入口差异化 |
| Cursor 内置 spec 管理 | 基础设施层吞噬 | OwlSpec 定位为方法论，不依赖特定 IDE |
| 新的 PM-AI 协作工具出现 | 验证市场存在 | 合作而非竞争 |
| AI 能力飞跃（AGI 级别） | spec 可能不再必要 | 短期（1-2 年）内不会发生 |
