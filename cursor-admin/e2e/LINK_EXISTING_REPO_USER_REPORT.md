## (b) 缺失与困惑点（真实用户视角）

1. **保存后无明确成功反馈（关联已有仓库）**  
   弹窗直接关闭，没有「项目已保存」或 Toast。用户难以确认是否保存成功，尤其在列表未立即出现新项目时。

2. **关联已有仓库缺少「下一步」说明**  
   选「关联已有仓库」时，没有文案说明保存后会发生什么（例如：符合工作目录规则的 Cursor 会话将归属到本项目），用户不知道后续要做什么。

3. **项目列表未出现新建项时无法区分原因**  
   若保存后列表仍空，用户无法判断是：创建失败、网络/后端错误，还是仅需刷新/等待。缺少错误提示或成功提示加剧困惑。

4. **项目详情入口依赖列表中存在项目名**  
   若列表因上述原因未展示新建项目，用户无法通过点击项目名进入项目详情，流程不闭环。

5. **项目参与 / 我的项目**  
   有表格和说明文案；空数据时的具体提示（如「暂无参与记录」）是否足够清晰，建议在无数据时再走查一遍。

## (c) 具体 UI/文案建议（让「关联已有仓库」流程更完整）

1. **保存成功反馈**  
   - 在「关联已有仓库」保存成功后：  
     - 先关闭弹窗并刷新列表；**且**  
     - 用 **Toast 或顶部/行内短提示** 显示「项目已保存」或「项目已创建」（约 2–3 秒消失）。  
   - 这样即使用户未立刻看到新行，也能确认操作成功。

2. **新建项目弹窗内「关联已有仓库」说明**  
   - 在「关联已有仓库」单选项下方增加一行说明，例如：  
     - 「保存后，在工作目录符合规则的 Cursor 会话将自动归属到本项目。」  
   - 可选：再补一句「成员需在对应目录下使用 Cursor，会话才会被计入。」  
   - 减少「保存完不知道接下来干嘛」的困惑。

3. **保存失败时的反馈**  
   - 若创建接口返回 4xx/5xx：  
     - **不要**仅关闭弹窗，应在弹窗内或 Toast 显示明确错误信息（如「保存失败：xxx」），并保留表单内容方便重试。

4. **空状态与列表**  
   - **项目参与**、**我的项目**：无数据时统一一句简短说明，例如「暂无参与记录」或「暂无项目数据」，并可与现有「数据来自 Git 采集 / Hook 上报」说明并存。  
   - **项目列表**：若保存成功但列表仍空，可在 Toast 成功文案后加「若未看到新项目，请刷新页面」；或列表上方提供「刷新」按钮并提示「刚刚已创建项目可点刷新查看」。

5. **项目详情的可达性**  
   - 确保「项目」列表在保存后正确刷新并展示新建项，以便用户能点击项目名进入项目详情，形成「新建 → 列表 → 详情」的完整路径。  
   - 若后端/环境导致列表暂时无数据，至少通过上述成功/失败反馈让用户知道当前状态。
