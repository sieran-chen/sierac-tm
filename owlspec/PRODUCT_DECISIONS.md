## PD-001：产品方向选择

**日期**: 2026-03-02
**状态**: 已决策

### 背景

从 OwlClaw 开发实践中发现，spec 驱动的半自动 AI 工程工作流具有独立产品化的潜力。

### 决策

**方法论 + 工具包。** 不做 SaaS 产品，不做 AI 编排器。

### 理由

1. 大厂已入场 spec 生成赛道（Kiro 免费、Spec Kit 开源），做 SaaS 没有价格优势
2. 方法论 + 培训/咨询的变现路径与 OwlClaw 完全一致，不分散精力
3. 开源方法论可以建立影响力，反哺 OwlClaw 的品牌认知

## PD-002：产品命名

**日期**: 2026-03-02
**状态**: 已决策

### 决策

**OwlSpec**

### 理由

- **品牌一致性**：OwlSpec（看清需求）→ OwlClaw（精准执行）→ OpenClaw（连接万物），形成完整链路
- **名称即定位**："Owl"= 在混沌中看清本质，"Spec"= 结构化规范
- **简短好记**：7 个字母，CLI 友好（`owlspec init`）
- **品牌故事**：猫头鹰先用锐利的眼睛看清猎物（OwlSpec），再用爪子精准捕获（OwlClaw）

## PD-003：核心定位修正

**日期**: 2026-03-02
**状态**: 已决策（第二版，基于行业研究修正）

### 背景

第一版定位是"Spec 驱动的 AI 工程方法论，面向技术开发者"。行业调研后发现这个定位有偏差。

### 第一版定位（已修正）

> "帮开发者写更好的 spec 来驱动 AI 编码"

**问题**：研发侧已有 Kiro/Spec Kit/OpenSpec，不缺 spec 工具。

### 第二版定位（当前）

> "从业务需求到 AI 交付的结构化桥梁——帮产品团队把模糊需求转化为 AI 能精准执行的规范"

### 修正依据

行业调研发现的关键事实：

1. **产品经理角色正在转型**：从"需求翻译官"变成"AI 指挥官"（人人都是产品经理，2026）
2. **研发侧反而更规范了**：SDD（Spec Driven Development）已成为主流范式（腾讯云、GitHub Blog）
3. **真正的瓶颈在产品侧**：PM 不知道怎么写"AI 能理解的需求"（VibeVibe.cn、CodingDiary）
4. **"氛围文档"质量参差不齐**：PM 的自然语言描述缺乏结构，导致 AI 反复猜测（多个实践案例）
5. **Vibe → Spec 的两阶段模式已成共识**：先 Vibe 探索方向，再 Spec 保证质量（CodingDiary 2026）

### 定位差异

| 维度 | 第一版 | 第二版 |
|------|--------|--------|
| 目标用户 | 技术开发者 | 产品经理 + 技术负责人 + 开发者 |
| 核心问题 | 开发者需要更好的 spec | PM 需要把需求"说清楚"给 AI |
| 价值主张 | 更好的 AI 编码工作流 | 业务需求到 AI 交付的结构化桥梁 |
| 差异化 | 多 Agent 协作（技术壁垒） | PM 侧需求结构化（市场空白） |

**日期**: 2026-03-02
**状态**: 已决策

### 决策

### 关系定义

```
OwlSpec（上游）→ 结构化 spec → AI 研发 → OwlClaw（下游）
```

- OwlClaw 开发优先（核心产品）
- OwlSpec 在 OwlClaw 开发过程中自然积累
- 文章/培训可以同时覆盖两个产品

## PD-008：业务表格作为需求起点

**日期**: 2026-03-02
**状态**: 已决策

### 背景

真实的企业业务需求往往不是从 PRD 开始的，而是从一张 Excel 表格开始的。采购经理的库存表、销售总监的客户跟进表、财务的预算表——这些表格里已经沉淀了数据模型、计算逻辑、业务规则。

### 决策

**将"业务表格解析"作为 OwlSpec 的第零层，作为核心差异化入口。**

### 理由

1. **降低起步门槛**：业务专家不需要学新工具，上传自己的 Excel 即可
2. **竞品空白**：没有任何竞品做"表格 → 结构化 spec"这件事
3. **天然的数据模型来源**：表格的列名就是字段，公式就是逻辑，条件格式就是规则
4. **解决"说清楚"的问题**：表格是业务专家最熟悉的表达方式，比让他们写 PRD 自然得多
5. **70/30 法则**：表格能提取约 70% 的业务规则，剩下 30% 通过交互式补充获取

### 实现路径

- **短期**：用 LLM（本地 Qwen / 云端 Claude）解析 Excel/CSV，输出 requirements.md 草稿
- **中期**：CLI 命令 `owlspec parse inventory.xlsx`
- **长期**：Web 界面上传表格 → 可视化编辑 spec

### 风险

- 表格解析精度有限 → 定位为"草稿"，人工审核是流程的一部分
- 复杂表格（多 Sheet、宏、VBA）→ 先支持简单表格，渐进扩展

## PD-009：两阶段模型（核心架构修正）

**日期**: 2026-03-02
**状态**: 已决策

### 背景

前两版把"业务分析"和"研发规范"混为同一层。实践思考后发现这是根本性错误——这两个阶段的角色、能力要求、工具、交付物完全不同。

### 决策

**明确分为两个阶段：**
- **阶段 1（业务分析）**：业务专家用 Skill + 云端大模型，输出数据流、信息流、HTML 线框图、业务规则清单
- **阶段 2（研发规范）**：研发专家用 Spec 三件套 + Spec 循环，输出 requirements.md / design.md / tasks.md

### 理由

1. **角色不同**：业务专家不懂技术，研发专家不懂业务细节——各做各擅长的
2. **工具不同**：阶段 1 用 Skill 就够了（不需要 IDE/Git），阶段 2 需要完整的开发工具链
3. **交付物不同**：阶段 1 输出业务文档（数据流、线框图），阶段 2 输出技术文档（spec 三件套）
4. **门槛不同**：阶段 1 零门槛（上传表格即可），阶段 2 需要架构能力
5. **行业验证**：6A 工作流（Align→Architect→Atomize→Approve→Automate→Assess）也是类似的分层思路

### 对产品的影响

- 阶段 1 的产品形态从"CLI 工具"简化为"一个 SKILL.md 文件"——更轻、更快、更容易分发
- Spec 三件套从"PM 写"修正为"研发写"——更符合实际
- 第一篇文章主题从"PM 如何说清楚"修正为"从 Excel 到 AI 系统"

## PD-010：阶段 1 交付物格式

**日期**: 2026-03-02
**状态**: 已决策

### 决策

- **数据流/信息流**：Mermaid 图（嵌入 Markdown）
- **线框图**：HTML（简单结构化 HTML，不是完整前端代码）
- **业务规则**：Markdown 表格
- **汇总文档**：Markdown

### 理由

| 格式 | AI 可读性 | 版本控制 | 业务专家可理解 |
|------|----------|---------|-------------|
| Mermaid | 极高（纯文本语法） | 极好 | 渲染后可读 |
| HTML | 高（结构化标签） | 好 | 浏览器可预览 |
| Markdown | 高 | 极好 | 人人可读 |

不用 Figma/PNG 的原因：AI 无法直接解析图片格式，且不可版本控制。

## PD-011：需求真源统一原则

**日期**: 2026-03-02
**状态**: 已决策（最高原则）

### 决策

**所有需求来自业务专家，业务专家是唯一需求真源。研发不翻译需求、不补充需求，只做技术决策。**

### 背景

前几版隐含了"业务专家只能做浅层分析"的假设。实践思考后发现：有了 AI 辅助，业务专家完全可以做到字段级的精确需求定义。

### 理由

1. **业务专家 + AI = 需求精度从"大概"提升到"字段级"**
2. 需求散落在多处是传统开发的根本问题——统一真源消除了这个问题
3. 研发"猜"业务意图是 bug 的主要来源——不猜就不会猜错
4. 变更控制有了清晰的权限边界——只有业务专家可以改需求

### 对产品的影响

- 阶段 1 的交付物从"草稿"升级为"权威定义"
- 阶段 2 的 requirements.md 从"研发重写"变为"引用需求真源"
- Skill 的核心价值从"替代分析"变为"帮业务专家做到足够深"
- 新增"需求真源冻结"和"变更控制"机制

## PD-012：表格在线化四步法（核心方法论修正）

**日期**: 2026-03-02
**状态**: 已决策（实践验证后的重大修正）

### 背景

用 v0.1 版 Skill 分析真实业务表格（Sierac 设备生产管理表单）时，产出了一个"大而全"的需求真源文档——20 条业务规则、14 个数据实体、MRP/齐套校验/采购预警等功能。看起来很专业，但业务专家反馈：**按这个逻辑做，很难落地，业务专家从一开始就会陷入困局。**

### 问题分析

v0.1 的 Skill 犯了两个根本错误：

1. **把调研文档当成了需求真源**：调研文档里很多是"人假定的场景"，不是已验证的业务逻辑。真正在跑的 Excel 表格才是需求真源。
2. **缺少方法论引导**：Skill 有"表格解析"能力，但没有教业务专家怎么思考。业务专家需要的不是一个能力清单，而是一个思维框架。

### 决策

**用"表格在线化四步法"替代原来的"五步工作流程"。**

四步法：
1. **看结构**：搞清楚表格管什么事、Sheet 之间什么关系
2. **拆实体**：从扁平的行列中识别出数据实体和关系（Excel 是扁平的，系统是结构化的）
3. **显规则**：把"人能看懂但系统不懂"的隐含信息找出来（缩写、颜色、空值含义、特殊值）
4. **定边界**：v1 = 表格在线化，不加不减，先跑起来

### 核心认知变化

| 维度 | 之前（v0.1） | 之后（v0.2） |
|------|------------|------------|
| 需求真源 | 调研文档 + 表格 | **只有正在跑的表格** |
| v1 目标 | 完整的 MOM/MES 系统 | **Excel 的在线化** |
| 方法论 | 能力清单（AI 能做什么） | **思维框架（业务专家怎么想）** |
| 产出节奏 | 一次性输出完整文档 | **每步确认，渐进深入** |
| 调研文档的角色 | v1 的需求输入 | **v2/v3 的演进参考** |

### 对 Skill 的影响

- Skill 从"能力导向"重构为"方法论导向"
- 新增"四步法"作为核心工作流程
- 新增"七种隐含规则"识别框架
- 新增"拆实体的三个信号"
- 删除了大量"需求文档解析"相关内容（调研文档不是 v1 的输入）
- 强调"每一步都要停下来等确认"

## PD-013：从表格到系统需要软件工程思维

**日期**: 2026-03-02
**状态**: 已决策

### 认知

从一张在跑的 Excel 到一个能跑的系统，中间有一个巨大的鸿沟。这个鸿沟需要软件工程思维来跨越：

1. **Excel 是扁平的，系统是结构化的**：一个 Sheet 里混着标题行、分区行、数据行、汇总行，需要拆成实体和关系
2. **Excel 是隐含的，系统是显式的**："外购芮"三个字人能看懂，系统需要拆成"来源类型"和"负责人"两个字段
3. **Excel 是展示结构，系统是逻辑结构**：表格的分区是展示，背后的工作令→部件→零件是逻辑

### 对 OwlSpec 的影响

OwlSpec 的 Skill 不只是"帮业务专家提取信息"，更重要的是**教业务专家用结构化的方式思考自己的表格**。这是 OwlSpec 区别于所有竞品的真正价值——不是 AI 能力，而是思维方法论。
