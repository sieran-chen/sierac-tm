# OwlSpec 方法论

> **版本**: v0.2.0 (2026-03-02)
> **状态**: 草案（第二版，增加产品侧视角）
> **一句话定义**: 从业务需求到 AI 交付的结构化桥梁——PM 在 spec 层决策，AI 在代码层执行。

---

## 〇、为什么需要 OwlSpec

### Vibe Coding 改变了什么

Vibe Coding（直觉式编程）让产品经理可以直接用自然语言描述需求，AI 生成代码。这打破了传统的 PRD → 评审 → 排期 → 开发 → 测试链条。

但 Vibe Coding 带来了新问题：

| 角色 | 传统模式的痛点 | Vibe Coding 的新痛点 |
|------|-------------|-------------------|
| **产品经理** | 写 PRD 费时、评审扯皮 | "AI 不懂我要什么"、需求描述不够精准 |
| **研发** | 被 PRD 绑架、需求变更频繁 | 反而更规范了（SDD），但 PM 给的输入质量差 |
| **团队** | 沟通成本高 | PM 和 AI 之间缺少"契约" |

### 行业现状（2026.3）

- **研发侧已有成熟方案**：Kiro（AWS）、Spec Kit（GitHub）、OpenSpec 做"spec → code"
- **产品侧缺少规范**：PM 写的"氛围文档"质量参差不齐，AI 反复猜测
- **协作层缺少标准**：PM 和 AI 研发团队之间没有共识的契约格式

### OwlSpec 的核心理念

> **不要让 AI 猜你想要什么，用 spec 告诉它。**
> **业务专家负责"做什么"和"为什么"，AI 负责"怎么做"。**
> **Spec 是业务和 AI 之间的契约，不是事后补的文档。**
> **业务表格是最自然的需求起点——让业务专家用自己熟悉的方式表达。**

## 一、第零层：业务表格解析

### 为什么从表格开始

真实的业务需求不是从 PRD 开始的，而是从一张 Excel 表格开始的。业务专家用表格管理日常工作，表格里已经沉淀了大量业务规则：

| 表格元素 | 蕴含的业务知识 | 提取为 |
|---------|-------------|--------|
| 列名 / 表头 | 数据模型 | requirements.md 中的数据字段定义 |
| 公式 | 计算逻辑 | requirements.md 中的业务规则 |
| 条件格式 | 业务约束 | requirements.md 中的验收标准 |
| 数据验证（下拉列表） | 枚举值 / 输入约束 | design.md 中的数据模型 |
| Sheet 间引用 | 数据关联 | design.md 中的表间关系 |
| 透视表 | 报表需求 | requirements.md 中的统计需求 |

### 工作流程

```
1. 业务专家提供表格（Excel / CSV / 飞书表格导出）
   ↓
2. OwlSpec AI 分析表格结构，生成 requirements.md 草稿
   - 识别数据模型（列名 → 字段定义）
   - 提取业务规则（公式 → 逻辑描述）
   - 发现约束条件（验证 → 边界定义）
   - 推断数据关系（引用 → 关联描述）
   ↓
3. 业务专家审核草稿，补充隐含规则
   （表格能提取 ~70% 的规则，剩下 30% 需要人工补充）
   ↓
4. 确认后进入三层文档结构
```

### 示例：库存管理表 → requirements.md

**输入**：一张库存管理 Excel

| 商品名称 | SKU | 库存数量 | 安全线 | 日均销量 | 状态 |
|---------|-----|---------|-------|---------|------|
| A 产品 | A001 | 50 | 100 | 20 | =IF(C2<D2,"预警","正常") |

**OwlSpec 提取的草稿**：

```markdown
## 数据模型
- 商品：名称、SKU（唯一标识）、库存数量、安全线、日均销量
- 状态：由库存数量与安全线的比较决定

## 业务规则
- 当库存数量 < 安全线时，状态为"预警"
- 否则状态为"正常"

## 待确认（需业务专家补充）
- [ ] 预警后的处理流程是什么？（自动通知？人工处理？）
- [ ] 日均销量的计算周期是多少天？
- [ ] 是否有"紧急"级别？阈值是多少？
- [ ] 补货建议量的计算公式是什么？
```

**业务专家补充后**：

```markdown
## 补充的隐含规则
- 预警后自动通知采购经理（邮件 + 企业微信）
- 日均销量 = 过去 30 天的平均值
- 库存 < 安全线的 50% 为"紧急"
- 建议补货量 = 日均销量 × 7 天（周五多算 3 天覆盖周末）
```

## 二、三层文档结构（第一层）

每个功能或重大变更必须有三层文档：

```
.specs/{feature-name}/
├── requirements.md    # 需求文档（WHAT — 做什么，PM 主笔）
├── design.md          # 设计文档（HOW — 怎么做，研发/AI 起草，PM 审核）
└── tasks.md           # 任务文档（WHEN — 什么顺序，AI 可辅助生成）
```

### 为什么是三层

| 层 | 谁写 | 解决什么问题 |
|----|------|------------|
| **requirements.md** | PM 主笔 | 防止 AI 猜需求——明确"做什么"和"不做什么" |
| **design.md** | 研发/AI 起草，PM 审核 | 防止 AI 乱选架构——明确技术约束和数据模型 |
| **tasks.md** | AI 辅助生成，人工审核 | 防止 AI 乱序执行——明确优先级和验收标准 |

大多数竞品只有 PRD + tasks 两层，缺少 design 层。**design 层是架构决策的载体**，没有它，AI 会在每次生成代码时做出不同的架构选择。

### requirements.md — PM 的主战场

PM 写 requirements.md 不需要懂代码，但需要做到：

1. **用户故事清晰**：作为 [谁]，我希望 [什么]，这样我可以 [价值]
2. **验收标准无歧义**：每个需求都有可检验的验收条件
3. **边界明确**：明确说"不做什么"，防止 AI 过度发挥
4. **业务规则显式化**：把隐含的业务规则写出来（AI 无法猜到）

```markdown
# Requirements: 库存预警

> **目标**：当商品库存低于安全线时自动提醒采购经理
> **优先级**：P0
> **预估工作量**：3-5 天

## 用户故事

**故事 1**：日常库存检查
- 作为采购经理，我希望每天早上 9 点收到库存预警
- 这样我可以及时补货，避免断货

**验收标准**：
- [ ] 每天 9:00 自动检查所有商品库存
- [ ] 库存低于安全线的商品列出：名称、当前库存、安全线、建议补货量
- [ ] 建议补货量 = 过去 30 天日均销量 × 7 天
- [ ] 周五的建议补货量多算 3 天（覆盖周末）
- [ ] A 类商品标记为"紧急"

## 不做什么（边界）

- 不自动下单，只提醒
- 不处理已停售商品
- 第一期不做多仓库支持
```

### design.md — 架构决策的载体

design.md 由研发或 AI 起草，PM 审核关键决策：

- 数据从哪来？（API/数据库/文件）
- 通知怎么发？（邮件/IM/短信）
- 异常怎么处理？（数据缺失/服务不可用）

PM 不需要看懂代码，但需要确认业务逻辑是否正确。

### tasks.md — 执行计划

tasks.md 可以由 AI 辅助生成，人工审核优先级和验收标准。

## 三、Spec 循环（第二层）

Spec 不是一次性写完就扔的文档，而是持续迭代的。

### 循环流程

```
1. Check — 检查未完成项，健康检查
   ↓
2. Plan — 选择本轮批次（1-3 个 task）
   ↓
3. Do — 对每个 task：
   ├─ Act（实现）
   ├─ Observe（测试）
   ├─ Reflect（反思）
   └─ Verify（验收）
   ↓
4. Doc — 更新进度，记录检查点
   ↓
5. Commit — 提交变更
   ↓
6. Evaluate — 是否继续？回到 1
```

### 核心规则

| 规则 | 说明 |
|------|------|
| 批次控制 | 每轮最多 3 个 task，避免上下文溢出 |
| 连续执行 | 一个 task 完成后，若下一个可执行，接着做 |
| 失败处理 | 连续失败 3 次 → 标记 blocked，人工介入 |
| 每轮提交 | 每轮结束必须 git commit |

## 四、多 Agent 协作（第三层）

当团队使用多个 AI Agent 并行开发时，需要物理隔离和协调机制。

### Worktree 隔离

```
project/              ← 主 worktree（统筹 + 人工）
project-review/       ← 审校 worktree（技术经理角色）
project-agent-1/      ← 编码 worktree 1
project-agent-2/      ← 编码 worktree 2
```

### 关键规则

- **各 Agent 只做分配给自己的 spec**，不越界
- **编码分支必须经过审校**才能合并
- **契约先行**：多端并行开发时，先输出共享契约（OpenAPI/JSON Schema）
- **每轮工作结束必须 commit**，工作目录保持干净

### 统筹循环

统筹者（人工或统筹 AI）定期执行：

```
Status → Merge → Sync → Assign → Unblock → Commit → Report
```

## 五、半自动决策模型

### 决策分层

| 层次 | 决策者 | 内容 |
|------|--------|------|
| 业务需求 | PM | 做什么、为谁做、优先级 |
| 架构设计 | 研发 + PM 审核 | 技术选型、数据模型 |
| 任务拆解 | AI + 人工审核 | 实施步骤、验收标准 |
| 代码实现 | AI | 编写代码、测试 |
| 质量审核 | AI + 人工 | spec 一致性、代码质量 |

### 为什么不全自动

全自动 AI 编码在以下场景失败：
- **隐含业务规则**：AI 无法猜到"周五多算 3 天"
- **架构决策**：AI 倾向于选最简单方案，不考虑长期维护
- **安全边界**：AI 可能引入安全漏洞
- **成本控制**：全自动可能在错误方向上消耗大量 token

## 六、迁移路径

从 Vibe Coding 到 OwlSpec 的渐进式迁移，提供两条入口：

### 入口 A：从表格开始（推荐给业务专家）

```
阶段 0：上传业务表格
  业务专家提供自己的 Excel → OwlSpec 解析 → 生成 requirements.md 草稿
  ↓
阶段 1：审核补充
  业务专家审核草稿，补充隐含规则（AI 无法从表格推断的部分）
  ↓
阶段 2：进入 Spec 循环
  AI 生成 design.md + tasks.md → 按循环推进开发
  ↓
阶段 3：多 Agent 协作（进阶）
  Worktree 隔离 + 审校 + 统筹
```

### 入口 B：从需求描述开始（推荐给 PM / 开发者）

```
阶段 0：Vibe Coding（现状）
  PM 在 Cursor 里和 AI 聊天，AI 直接写代码
  ↓
阶段 1：加 tasks.md（最小改动，1 天）
  每个功能写一个 tasks.md，列出要做的事和验收标准
  ↓
阶段 2：加 requirements.md（PM 参与，3 天）
  PM 写结构化需求，AI 不再猜需求
  ↓
阶段 3：三层完整 + Spec 循环（1 周）
  加 design.md，按循环持续推进
  ↓
阶段 4：多 Agent 协作（进阶）
  Worktree 隔离 + 审校 + 统筹
```

**入口 A 的门槛更低**——业务专家不需要学任何新东西，只需要提供自己已有的表格。

## 七、适用场景

| 场景 | 适用度 | 推荐入口 | 建议 |
|------|--------|---------|------|
| 业务专家想把表格变成系统 | 极高 | 入口 A | 表格解析 → 三层 spec → AI 开发 |
| 生产级软件开发 | 高 | 入口 B | 完整三层 + Spec 循环 |
| PM 用 AI 做 MVP | 高 | 入口 A 或 B | 至少 requirements.md + tasks.md |
| 多人/多 Agent 协作 | 高 | 入口 B | 完整四阶段 |
| 快速原型/hackathon | 低 | — | 直接 Vibe Coding |
| 一次性脚本 | 低 | — | 不需要 spec |
