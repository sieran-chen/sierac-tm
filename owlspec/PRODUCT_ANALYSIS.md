# OwlSpec 产品分析

> **创建时间**: 2026-03-02
> **最后更新**: 2026-03-02（第二版，基于行业研究重新审视）
> **来源**: Cursor 对话讨论 + 行业调研

## 一、重新审视：真正的问题是什么

### 第一版的认知（已修正）

第一版分析的出发点是"开发者需要更好的 spec 来驱动 AI 编码"。这个视角是**技术人的自嗨**——只看到了开发侧的痛点。

### 行业调研后的修正

从人人都是产品经理、腾讯云开发者社区、GitHub Blog 等渠道的调研显示，2026 年的真实图景是：

**传统开发流程正在被 Vibe Coding 颠覆，但颠覆的不是"开发"环节，而是"产品→研发"的协作模式。**

| 传统模式 | Vibe Coding 模式 | 真正的痛点 |
|---------|-----------------|-----------|
| PM 写 PRD → 评审 → 排期 → 开发 → 测试 | PM 描述需求 → AI 直接生成代码 | PM 不知道怎么"说清楚"给 AI |
| 研发是"翻译官"，把 PRD 翻译成代码 | 研发变成"架构师 + 审核官" | 研发反而更规范了（SDD/Spec Coding） |
| PM 和研发之间有大量沟通成本 | PM 可以直接用 AI 验证想法 | 但 PM 产出的"氛围文档"质量参差不齐 |

### 核心洞察

> **Vibe Coding 时代，研发侧已经有了 SDD（Spec Driven Development）的成熟方案（Kiro、Spec Kit、OpenSpec）。
> 真正缺失的是：产品侧如何把模糊的业务需求，结构化地传递给 AI 驱动的研发流程。**

换句话说：
- **研发侧**：已经有 Kiro/Spec Kit/OpenSpec 做"spec → code"
- **产品侧**：缺少"业务需求 → 结构化 spec"的规范和工具
- **协作层**：缺少"PM 和 AI 研发团队之间的契约标准"

## 二、被忽视的真实起点：业务表格

### 需求不是从 PRD 开始的

之前的分析假设需求的起点是"PM 脑中的想法"或"一段自然语言描述"。但在真实的企业场景中：

> **业务需求往往从一张 Excel 表格开始。**

采购经理有一张库存表，销售总监有一张客户跟进表，财务有一张预算表。这些表格里**已经沉淀了业务规则**——列名是数据模型，公式是计算逻辑，条件格式是业务规则，筛选条件是查询需求。

### 当前的断裂

```
业务表格（Excel/飞书/钉钉表格）
    │
    │ ← 【断裂 1：表格里的业务规则没人提取】
    ↓
PM 的脑子（模糊的需求）
    │
    │ ← 【断裂 2：PM 不知道怎么结构化给 AI】
    ↓
结构化 Spec
    │
    │ ← Kiro / Spec Kit / OpenSpec（已覆盖）
    ↓
AI 生成代码 → 可运行软件
```

**两个断裂点**：
1. **表格 → 需求**：业务专家用表格管理业务，但没人帮他们从表格中提取出"这个表格背后的业务规则是什么"
2. **需求 → Spec**：即使 PM 知道需求，也不知道怎么结构化给 AI

### 现有工具只解决了表面问题

| 工具 | 做了什么 | 没做什么 |
|------|---------|---------|
| **ChatExcel / 表答** | 用自然语言操作表格（公式、图表） | 不提取业务规则，不生成 spec |
| **Excelmatic / Quadratic** | AI 驱动的表格分析 | 分析数据，不分析业务逻辑 |
| **勤哲 Excel 服务器** | Excel → 管理系统（零代码） | 绑定自家平台，不输出通用 spec |
| **PM33 / Omniflow** | AI 生成 PRD | 从描述生成，不从表格生成 |
| **CodeGuide** | idea → PRD/tech spec | 一次性生成，无迭代 |
| **Kiro** | 自然语言 → EARS → design → tasks | 面向开发者，PM 看不懂 EARS |

**没有一个工具做"业务表格 → 结构化 spec"这件事。**

### 这才是 OwlSpec 的真正起点

```
业务表格（Excel/飞书表格）
    │
    │ ← OwlSpec 的第零层：表格解析 + 业务规则提取
    ↓
结构化业务需求（requirements.md）
    │
    │ ← OwlSpec 的第一层：需求结构化
    ↓
架构设计（design.md）+ 任务清单（tasks.md）
    │
    │ ← OwlSpec 的第二层：Spec 循环
    ↓
AI 生成代码 → 可运行软件
```

## 三、市场格局（含表格场景）

### 竞品在"表格 → Spec"这段做了什么

| 产品 | 做了什么 | 缺什么 |
|------|---------|--------|
| **ChatExcel** | 自然语言操作表格 | 不提取业务规则，不生成 spec |
| **Quadratic** | AI 表格分析（连接多数据源） | 分析数据不分析业务逻辑 |
| **勤哲 Excel 服务器** | Excel → 管理系统 | 闭源平台，不输出通用 spec |
| **CodeGuide** | idea → PRD/tech spec | 不接受表格输入 |
| **Kiro** | 自然语言 → EARS → design → tasks | 不接受表格输入，面向开发者 |
| **PM33** | AI 生成 PRD + 优先级评分 | 不接受表格输入 |

### 真正的空白

**没有一个产品做到**：
1. 从业务表格中**提取隐含的业务规则**（列名→数据模型，公式→计算逻辑，条件→业务约束）
2. 将提取的规则**转化为 AI 能理解的结构化 spec**
3. 让业务专家**在自己熟悉的表格语境中**审核和补充需求
4. 结构化 spec 无缝传递给 AI 研发流程

## 四、OwlSpec 的重新定位

### 旧定位（已修正）

> "Spec 驱动的 AI 工程方法论" — 面向技术开发者

### 新定位

> **OwlSpec：从业务需求到 AI 交付的结构化桥梁**
>
> 帮助产品团队把模糊的业务需求，转化为 AI 研发团队能精准执行的结构化规范。

### 为什么叫 OwlSpec

猫头鹰的核心能力是**在黑暗中看清猎物**。OwlSpec 的核心能力是**在模糊的业务需求中看清结构**。

- **Owl**（猫头鹰）= 在混沌中看清本质
- **Spec**（规范）= 结构化的、可执行的、无歧义的

### 品牌体系

```
OwlSpec  — 看清需求：业务需求 → 结构化规范（产品侧 + 协作层）
OwlClaw  — 精准执行：结构化规范 → AI Agent 自主决策（研发侧 + 业务系统）
OpenClaw — 连接万物：AI 助理 + IM 入口（沟通层）
OwlHub   — 知识聚集：Skills 生态市场（生态层）
```

## 五、OwlSpec 的四层价值

### 第零层：业务表格解析（面向业务专家）

> 这是 OwlSpec 区别于所有竞品的核心差异化。

业务专家（采购、销售、财务、运营）不写 PRD，他们用表格管理业务。OwlSpec 从表格出发：

| 表格元素 | 提取为 | 示例 |
|---------|--------|------|
| 列名 / 表头 | 数据模型字段 | "商品名称、库存数量、安全线" → `Product.name, Product.stock, Product.safety_threshold` |
| 公式 | 计算逻辑 / 业务规则 | `=IF(B2<C2, "预警", "正常")` → "库存低于安全线时触发预警" |
| 条件格式 | 业务约束 | 红色标记 < 100 → "库存低于 100 为紧急" |
| 数据验证 | 输入约束 | 下拉列表 → 枚举类型 |
| 筛选/排序 | 查询需求 | 按部门筛选 → "需要按部门查看" |
| Sheet 间引用 | 数据关联 | VLOOKUP → 表间关系 |
| 透视表 | 报表需求 | 按月汇总 → "需要月度统计报表" |

**工作流程**：

```
1. 业务专家上传/粘贴表格
   ↓
2. OwlSpec AI 分析表格结构，提取：
   - 数据模型（列名 → 字段）
   - 业务规则（公式 → 逻辑）
   - 约束条件（验证 → 边界）
   - 数据关系（引用 → 关联）
   ↓
3. 生成 requirements.md 草稿（人类可读）
   ↓
4. 业务专家审核、补充隐含规则
   （"周五的补货量要多算 3 天"——这种规则表格里没有，但业务专家知道）
   ↓
5. 确认后进入 Spec 循环
```

**关键洞察**：表格能提取 70% 的业务规则，但剩下 30% 的隐含规则（行业惯例、特殊场景、例外处理）只有业务专家知道。OwlSpec 的价值是**用表格解析降低起步门槛，用交互式补充捕获隐含规则**。

### 第一层：需求结构化规范（面向 PM）

帮 PM 把业务需求（可能来自表格解析，也可能来自口头描述）转化为 AI 能精准理解的三层文档：

| 文档 | PM 写什么 | AI 用来做什么 |
|------|---------|-------------|
| **requirements.md** | 用户故事、验收标准、业务规则 | 理解"做什么" |
| **design.md** | 架构约束、数据模型、API 契约 | 理解"怎么做"（PM 审核，研发/AI 起草） |
| **tasks.md** | 优先级、阶段划分、验收检查 | 理解"按什么顺序做" |

**关键**：PM 不需要懂代码，但需要懂"如何无歧义地描述业务规则"。OwlSpec 提供模板和规范，降低这个门槛。如果有表格输入，第零层已经帮 PM 完成了大部分工作。

### 第二层：Spec 循环（面向研发团队）

结构化 spec 写好后，研发团队（人 + AI Agent）按 spec 循环推进：

```
Check → Plan → Do(Act/Observe/Reflect/Verify) → Doc → Commit → Evaluate
```

这一层是从 OwlClaw 开发实践中验证的，包括：
- 批次控制（每轮 1-3 个 task）
- 健康检查（连续失败自动标记 blocked）
- 每轮提交（可追溯）

### 第三层：多 Agent 协作（面向技术负责人）

当团队使用多个 AI Agent 并行开发时：
- Worktree 物理隔离
- 审校 Loop（技术经理角色）
- 统筹循环（merge/分配/同步）
- 契约先行规则

## 六、目标用户重新定义

### 核心用户：业务专家 / 产品经理

| 画像 | 痛点 | OwlSpec 给什么 |
|------|------|---------------|
| 用 Excel 管理业务的部门负责人 | "我的表格能变成系统吗？" | 表格解析 → 结构化 spec → AI 开发 |
| 用 Cursor/Claude Code 做 MVP 的 PM | "AI 不懂我要什么" | 需求结构化模板 + 验收标准写法 |
| 带 AI 研发团队的 Tech Lead | "PM 需求太模糊，AI 乱猜" | PM-研发之间的 spec 契约标准 |
| 想引入 AI 编码的传统团队 | "不知道怎么适配 AI" | 从 Vibe Coding 到 Spec Engineering 的迁移路径 |

### 次要用户：独立开发者 / 技术专家

| 画像 | 痛点 | OwlSpec 给什么 |
|------|------|---------------|
| 一个人用 AI 做项目 | 代码越改越烂、架构漂移 | 三层文档 + Spec 循环 |
| 管理多个 AI Agent 的技术专家 | 多 Agent 互相覆盖 | Worktree 隔离 + 统筹循环 |

## 七、产品形态

### 短期（0-3 个月）：方法论 + 模板 + 表格解析 PoC

1. **方法论文档**（开源）— OwlSpec Methodology
2. **模板仓库**（开源）— `owlspec/owlspec-template`
3. **表格解析 PoC** — 用 LLM 解析 Excel/CSV，输出 requirements.md 草稿
4. **文章系列** — "Vibe Coding 时代，产品经理如何'说清楚'给 AI"

### 中期（3-6 个月）：CLI 工具

```bash
owlspec init                    # 初始化项目 spec 结构
owlspec new feature-name        # 创建新 spec（三层文档骨架）
owlspec parse inventory.xlsx    # 解析业务表格 → requirements.md 草稿
owlspec check                   # 健康检查（文档完整性 + 进度）
owlspec status                  # 全局进度报告
owlspec worktree init           # 创建多 Agent worktree 结构
```

### 长期（6-12 个月）：根据反馈决定

- 如果业务专家用户多 → 做 Web 界面（上传表格 → 可视化编辑 spec）
- 如果开发者用户多 → 做 IDE 插件（Cursor/VS Code 集成）
- 如果培训需求多 → 做课程体系（"从 Excel 到 AI 系统"）

## 八、与 OwlClaw 的关系（修正版）

```
业务专家的 Excel 表格
    │
    │ OwlSpec 表格解析
    ↓
产品经理 / 业务负责人 审核补充
    │
    │ OwlSpec 需求结构化
    ↓
OwlSpec（结构化 spec + Spec 循环 + 多 Agent 协作）
    │
    │ spec 文档作为契约传递
    ↓
AI 研发团队（Cursor / Claude Code / Codex CLI）
    │
    │ 开发出的业务系统
    ↓
OwlClaw（让业务系统获得 AI 自主能力）
    │
    │ SKILL.md 是 spec 的特化形式
    ↓
AI Agent 自主运行业务流程
```

**OwlSpec 是上游（表格/需求 → 代码），OwlClaw 是下游（代码 → AI 自主运行）。**

## 九、风险重新评估

| 风险 | 严重度 | 缓解 |
|------|--------|------|
| PM 不愿意学新规范 | 中→低 | 表格解析大幅降低起步门槛——业务专家不用学新东西，上传表格即可 |
| 大厂做类似的事 | 高 | Kiro 面向开发者不面向业务专家；表格解析是我们的独特入口 |
| 方法论变现难 | 中 | 培训/咨询 + OwlClaw 的商业化路径共享 |
| AI 进步让 spec 不再必要 | 低 | 短期内 AI 仍无法理解隐含业务规则，spec 是必要的 |
| OpenSpec 加入 PM 侧功能 | 中 | 先发优势 + 中文原生 + 表格入口差异化 |
| 表格解析精度不够 | 中 | 定位为"草稿"而非"成品"，人工审核补充是流程的一部分 |
