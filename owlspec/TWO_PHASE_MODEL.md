# OwlSpec 两阶段模型：需求真源 → 技术实现

> **创建时间**: 2026-03-02
> **状态**: 核心架构文档（v3.1，增加"需求真源统一"原则）
> **核心原则**: 所有需求来自业务专家，业务专家是唯一需求真源。研发不翻译需求、不补充需求，只做架构决策和代码实现。

---

## 〇、最高原则：需求真源统一

> **所有的需求都来自于业务专家。业务专家可以做到很深。**

这条原则决定了整个模型的重心：

| 传统模式 | OwlSpec 模式 |
|---------|-------------|
| 业务说个大概，研发去"翻译"和"补充" | **业务专家把需求做到极致细粒度，研发只做技术决策** |
| 需求散落在 PRD、会议记录、口头沟通中 | **需求统一沉淀在业务专家的交付物中，这是唯一真源** |
| 研发经常"猜"业务意图 | **研发不猜——需求文档里没有的，就是不做的** |
| 需求变更时 PM 和研发扯皮 | **变更只能由业务专家发起，修改唯一真源** |

**为什么业务专家可以做到很深？**

因为有了 AI 的辅助，业务专家不再受限于"不懂技术所以只能说个大概"。借助云端大模型的高级能力，业务专家可以：

- 把一张 Excel 表格拆解成完整的数据模型和业务规则
- 把一个审批流程拆解成精确的状态机和异常处理
- 把一个模糊的"我想要个看板"拆解成具体的页面结构、数据字段、交互逻辑
- 把隐含的行业惯例显式化为可检验的业务规则

**AI 赋能了业务专家，让他们的需求表达能力从"说个大概"提升到"精确到字段级"。这才是 OwlSpec 的真正价值。**

## 一、前几版的错误

### 第一版错误：只看到研发侧

> "帮开发者写更好的 spec" — 忽略了业务侧

### 第二版错误：把两个阶段混在一起

> "从业务表格到 AI 交付的结构化桥梁" — 方向对了，但把"业务分析"和"研发规范"混为同一层

### 第三版错误：低估了业务专家

> "表格能提取 70% 的规则，剩下 30% 需要人工补充" — 暗示业务专家只能做浅层分析

### v3.1 修正：业务专家是需求的唯一权威

业务专家不是"提供初步输入然后交给研发深化"的角色。业务专家是**需求的唯一权威**——他们的交付物是完整的、深入的、不可被研发覆盖的。

```
阶段 1：需求定义（业务专家 — 唯一需求真源）
  输入：业务表格、流程文档、现有系统截图、行业知识
  工具：云端大模型（Claude/GPT）+ OwlSpec Skill
  输出：完整的需求真源文档（数据流、信息流、线框图、业务规则、验收标准）
  深度：字段级数据模型、完整业务规则、异常场景、边界条件
  ──────────────────────────────────────
  需求真源冻结（变更只能由业务专家发起）
  ──────────────────────────────────────
阶段 2：技术实现（研发专家 — 只做技术决策）
  输入：阶段 1 的需求真源（只读，不可修改业务规则）
  工具：Cursor/Claude Code + OwlSpec Spec 循环
  输出：spec 三件套（requirements.md / design.md / tasks.md）
  权限：可以决定"怎么实现"，不可以决定"做什么"
```

**研发不翻译需求，不补充需求。需求真源里没有的，就是不做的。**

## 二、阶段 1：需求定义（业务专家 — 唯一需求真源）

### 谁来做

业务专家 / 产品经理 / 业务分析师 — 懂业务，借助 AI 可以做到极深。

### 业务专家能做到多深

传统认知中，业务专家"说个大概"，研发去"翻译细化"。但有了 AI 辅助后：

| 传统深度 | AI 辅助后的深度 |
|---------|---------------|
| "我要一个库存预警功能" | 完整的数据模型（字段名、类型、约束）、计算公式、触发条件、异常处理 |
| "审批流程大概是这样的" | 精确的状态机（每个状态、转换条件、超时处理、回退规则） |
| "我想要个看板" | 具体的页面结构、每个区块的数据来源、筛选条件、排序规则、交互逻辑 |
| "这个行业一般是这样做的" | 显式化的业务规则清单，每条规则有来源、优先级、验收标准 |

**业务专家 + AI = 需求精度从"大概"提升到"字段级"。** 这是 OwlSpec 存在的根本前提。

### 输入是什么

| 输入类型 | 示例 | 蕴含的业务知识 |
|---------|------|-------------|
| 业务表格 | Excel 库存表、客户跟进表 | 数据模型、计算逻辑、业务规则 |
| 流程文档 | Word/PDF 审批流程、操作手册 | 业务流程、角色权限、异常处理 |
| 现有系统截图 | ERP 界面、报表截图 | 页面结构、数据展示、交互模式 |
| 行业知识 | 业务专家的经验和判断 | 隐含规则、行业惯例、例外场景 |

### 用什么工具

**核心洞察：这个阶段用 Skill 就能快速实现。**

为什么？因为：
1. 云端大模型已经具备强大的文档分析和结构化能力
2. 业务分析的输出是**文档**，不是代码——不需要 IDE，不需要 Git
3. 一个精心设计的 Skill 引导大模型**向业务专家提出正确的问题**，帮助业务专家把隐含知识显式化

```
Skill: OwlSpec Business Analyzer
  ├── 表格解析：上传 Excel → 提取数据模型 + 业务规则 + 计算逻辑
  ├── 流程分析：上传流程文档 → 提取状态机 + 数据流 + 信息流
  ├── 深度追问：基于分析结果 → 向业务专家追问边界条件和异常场景
  ├── 线框图生成：基于完整需求 → 生成 Mermaid/HTML 线框图
  └── 需求真源组装：汇总所有分析 → 输出完整的需求真源文档
```

**Skill 的核心价值不是"替代业务专家分析"，而是"帮业务专家把脑子里的东西完整地、结构化地表达出来"。** AI 是业务专家的放大器，不是替代者。

### 交付物是什么（需求真源）

| 交付物 | 格式 | 性质 |
|--------|------|------|
| **数据模型** | Mermaid ER 图 + Markdown 字段表 | **权威定义**：字段名、类型、约束、关系 |
| **数据流图** | Mermaid flowchart | **权威定义**：数据从哪来、怎么变换、到哪去 |
| **信息流图** | Mermaid 时序图 | **权威定义**：角色间的信息传递和触发条件 |
| **线框图** | HTML 或 Mermaid | **权威定义**：页面结构、交互逻辑 |
| **业务规则清单** | Markdown 表格 | **权威定义**：每条规则有来源、优先级、验收标准 |
| **需求真源文档** | Markdown（汇总以上所有） | **唯一真源**：研发不可修改业务规则部分 |

注意"权威定义"——这不是"草稿"或"初步分析"。这是**最终的、完整的、不可被研发覆盖的需求定义**。

### 为什么线框图用 HTML/Mermaid 而不是 Figma

| 格式 | AI 可读性 | 版本控制 | 业务专家可编辑 | 研发可消费 |
|------|----------|---------|-------------|----------|
| Figma | 低（需 API） | 差 | 需学工具 | 需导出 |
| PNG/截图 | 低（需 OCR） | 差 | 不可编辑 | 只能看 |
| **HTML** | **高（结构化）** | **好（文本）** | 可用 AI 辅助 | **直接可用** |
| **Mermaid** | **极高（语法化）** | **极好（纯文本）** | 可用 AI 辅助 | **直接可用** |

**关键**：线框图的目的不是"好看"，而是"让下游（研发/AI）精准理解页面结构和交互逻辑"。HTML 和 Mermaid 是最 AI 友好的格式。

## 三、阶段 2：技术实现（研发专家 — 只做技术决策）

### 谁来做

研发专家 / Tech Lead / 高级开发者 — 懂架构，懂 AI 编码工具。

### 权限边界（关键）

| 研发可以决定 | 研发不可以决定 |
|------------|-------------|
| 用什么技术栈 | 做什么功能 |
| 数据库表结构怎么设计 | 有哪些业务规则 |
| API 接口怎么设计 | 验收标准是什么 |
| 代码架构怎么组织 | 优先级是什么 |
| 性能怎么优化 | 用户交互逻辑是什么 |

**如果研发在实现过程中发现需求有歧义或遗漏，正确的做法是：回到业务专家那里确认，而不是自己猜。** 需求真源只有一个。

### 输入是什么

阶段 1 的需求真源文档——完整的、权威的、不可被研发修改（业务规则部分）的需求定义。

### 用什么工具

Cursor / Claude Code / Codex CLI + OwlSpec Spec 循环方法论。

### 输出是什么（Spec 三件套）

| 文档 | 内容 | 与需求真源的关系 |
|------|------|---------------|
| **requirements.md** | 用户故事 + 验收标准 + 业务规则 | **直接引用**需求真源，不重写、不改写 |
| **design.md** | 架构决策 + 数据模型 + API 契约 + 技术约束 | 研发的**技术决策**，基于需求真源的数据模型和数据流 |
| **tasks.md** | 任务拆解 + 优先级 + 验收检查 | 研发的**实施计划**，优先级来自需求真源 |

### requirements.md 的特殊性

requirements.md 不是研发"重新写"的需求文档。它是：

- **业务规则部分**：直接从需求真源复制或引用，研发不可修改
- **技术约束部分**：研发补充（性能要求、安全要求、兼容性要求）
- **验收标准部分**：直接从需求真源的业务规则清单转化，研发不可降低标准

### 为什么 Spec 三件套在阶段 2

| 维度 | 需求真源（阶段 1） | Spec 三件套（阶段 2） |
|------|-----------------|-------------------|
| 写给谁看 | 业务专家自己审核 + 研发消费 | AI Agent 执行 |
| 语言 | 业务语言（"库存低于安全线"） | 技术语言（"stock < safety_threshold"） |
| 决策权 | 业务决策（做什么） | 技术决策（怎么做） |
| 可修改性 | 只有业务专家可以修改 | 研发可以修改技术部分 |
| 数据模型 | 概念级（字段名、业务含义、关系） | 物理级（表结构、类型、索引、分区） |

## 四、需求真源冻结与变更控制

### 冻结检查清单

阶段 1 完成后，需求真源进入"冻结"状态。冻结前必须满足：

- [ ] **数据模型完整**：所有业务实体、字段、关系已定义
- [ ] **业务规则无遗漏**：每条规则有来源标注、优先级、验收标准
- [ ] **数据流完整**：数据从哪来、怎么变换、到哪去
- [ ] **信息流完整**：角色之间的信息传递、触发条件、异常处理
- [ ] **线框图覆盖核心页面**：主要交互页面有线框图
- [ ] **边界明确**：明确"不做什么"
- [ ] **异常场景已覆盖**：业务专家确认了所有异常处理方式
- [ ] **业务专家签字确认**：这是最终版本，可以进入研发

### 变更控制

冻结后，如果需要修改需求：

1. **只有业务专家可以发起变更**——研发不可以自行修改业务规则
2. **变更必须更新需求真源文档**——不允许口头变更
3. **变更影响评估**——研发评估技术影响，但业务决策权在业务专家
4. **版本记录**——每次变更记录原因、影响范围、决策人

### 需求真源文件结构

```
{feature-name}/
├── SOURCE.md            # 需求真源文档（汇总，唯一权威）
├── data-model.md        # 数据模型（Mermaid ER 图 + 字段表）
├── data-flow.md         # 数据流图（Mermaid）
├── info-flow.md         # 信息流图（Mermaid 时序图）
├── wireframes/          # 线框图
│   ├── dashboard.html   # 看板页面
│   └── detail.html      # 详情页面
├── business-rules.md    # 业务规则清单（每条有来源和验收标准）
└── CHANGELOG.md         # 变更记录
```

## 五、OwlSpec 产品在两阶段中的角色

### 阶段 1 的工具：OwlSpec Business Analyzer Skill

**形态**：一个 Skill 文件（SKILL.md），可以在 Claude/Cursor/Codex 中使用。

**核心设计理念**：Skill 不是替代业务专家思考，而是**帮业务专家把脑子里的东西完整地、结构化地表达出来**。AI 是放大器，业务专家是决策者。

**能力**：
1. **表格解析**：上传 Excel → 提取数据模型 + 业务规则 + 计算逻辑
2. **流程分析**：上传流程文档 → 提取状态机 + 数据流 + 信息流
3. **深度追问**：基于分析结果 → 向业务专家追问边界条件、异常场景、行业惯例
4. **线框图生成**：基于完整需求 → 生成 Mermaid/HTML 线框图
5. **需求真源组装**：汇总所有分析 → 输出完整的需求真源文档（含冻结检查清单）

**为什么 Skill 就够了**：
- 阶段 1 的输出是文档，不是代码
- 云端大模型已有足够的分析能力
- Skill 的核心价值是**引导业务专家做到足够深**——通过结构化的追问，确保没有遗漏
- 不需要额外的 CLI 工具或 Web 平台

### 阶段 2 的工具：OwlSpec Spec 循环方法论 + 模板

**形态**：方法论文档 + 模板仓库 + AGENTS.md 规则。

**能力**：
1. **Spec 三件套模板**：从阶段 1 交付物快速生成 requirements/design/tasks
2. **Spec 循环**：Check → Plan → Do → Doc → Commit → Evaluate
3. **多 Agent 协作**：Worktree 隔离 + 审校 + 统筹
4. **健康检查**：连续失败检测、进度追踪

## 七、这个模型解决了什么问题

| 传统问题 | OwlSpec 两阶段模型的解决 |
|---------|----------------------|
| 需求散落在各处，没有唯一真源 | **需求真源统一**——所有需求沉淀在业务专家的交付物中 |
| 研发"猜"业务意图，猜错了扯皮 | **研发不猜**——需求真源里没有的就是不做的 |
| 业务专家"说个大概"，细节靠研发补 | **业务专家做到很深**——AI 辅助下可以精确到字段级 |
| PM 不愿意学 spec 三件套 | **PM 不需要学**——阶段 1 用 Skill 引导，输出的是业务文档 |
| 需求变更时 PM 和研发扯皮 | **变更控制**——只有业务专家可以修改需求真源 |
| 研发拿到的需求太模糊 | **需求真源已经极其完整**——数据模型、业务规则、线框图全有 |
| 工具太重，门槛太高 | **阶段 1 只需要一个 Skill**，不需要安装任何东西 |

## 八、产品路线图

### 短期（0-3 个月）

1. **OwlSpec Business Analyzer Skill**（阶段 1 核心）
   - 一个 SKILL.md 文件
   - 表格解析 + 流程分析 + 深度追问 + 线框图生成 + 需求真源组装
   - 冻结检查清单
   - 在 Claude/Cursor 中可直接使用
   - **用自己的真实业务场景验证**

2. **OwlSpec 方法论文档**（阶段 2 指南）
   - 更新后的方法论（含需求真源统一原则）
   - Spec 三件套模板（含需求真源引用规范）

3. **第一篇文章**
   - "AI 让业务专家做到很深：从 Excel 到完整需求真源"

### 中期（3-6 个月）

4. **模板仓库** `owlspec/owlspec-template`
   - 阶段 1 需求真源文件结构模板
   - 阶段 2 spec 三件套模板（含需求真源引用）
   - AGENTS.md 规则（含"不可修改需求真源"约束）
   - 多 Agent worktree 脚本

5. **行业 Skill 变体**（如果有需求）
   - 电商/零售版（库存、订单、客户管理）
   - 制造/供应链版
   - 金融/财务版

### 长期（6-12 个月）

根据反馈决定：
- 阶段 1 用户多 → 做更多行业 Skill + 培训
- 阶段 2 用户多 → 做 IDE 插件
- 企业需求多 → 做需求真源管理平台
